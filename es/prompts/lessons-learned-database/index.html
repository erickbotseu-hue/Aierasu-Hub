Consejo Pro: Optimiza tus resultados iterando el prompt.

Act as a continuous improvement specialist who designs systems for capturing institutional knowledge — because the most expensive lessons are the ones you pay for twice. Create a lessons learned database template for ongoing projects. INPUTS: - Organization/team: [WHO THIS IS FOR] - Project types: [WHAT KINDS OF PROJECTS YOU RUN] - Current knowledge capture: [HOW YOU DOCUMENT LESSONS NOW — or "we don't"] - Access needs: [WHO SHOULD BE ABLE TO SEARCH AND CONTRIBUTE] LESSONS LEARNED DATABASE: 1. **Entry Template:** | Field | Content | - Lesson ID: [AUTO-INCREMENT] - Date captured: [DATE] - Project/context: [WHICH PROJECT] - Category: [TECHNICAL / PROCESS / PEOPLE / VENDOR / CLIENT / SCOPE] - What happened: [FACTUAL DESCRIPTION] - What we learned: [THE INSIGHT — in plain language] - What to do differently: [SPECIFIC ACTION FOR FUTURE PROJECTS] - Impact: [COST / TIME / QUALITY IMPACT OF THE ORIGINAL ISSUE] - Contributor: [WHO SUBMITTED THIS LESSON] - Tags: [SEARCHABLE KEYWORDS] 2. **Categories and Examples:** | Category | Example Lesson | - Estimation: "We underestimated API integration by 2x — always add buffer for third-party dependencies" - Communication: "Client didn't see the prototype until Week 6 — show mockups by Week 2" - Technical: "Migration scripts must be tested on production-size data, not samples" - Process: "Daily standups with 10+ people become status reports — split into sub-teams" 3. **Collection Process:** | When | How | Owner | - After each project: post-mortem generates entries - During projects: team can submit anytime via form - Monthly: team meeting reviews recent lessons - Quarterly: leadership reviews patterns and systemic issues 4. **Retrieval Process:** | Trigger | Action | - Starting new project → search database for relevant categories - Project kickoff → share top 5 relevant lessons from past projects - Risk assessment → cross-reference known risks from database - New team member → assign reading of top 10 organizational lessons 5. **Review and Maintenance:** | Action | Frequency | - Tag cleanup and deduplication: quarterly - Relevance review: annually (archive outdated lessons) - Pattern analysis: quarterly (look for repeated issues) - Impact measurement: did implementing lessons prevent problems? CONSTRAINTS: - Make submission EASY — complex forms don't get filled out - Lessons must be ACTIONABLE — "communication is important" isn't a lesson; "Send weekly status updates to clients" IS - No blame in entries — focus on WHAT HAPPENED and WHAT TO DO, not WHO - SEARCH must work well — a database nobody can search is a database nobody uses - Actually USE the database — require project managers to check it at kickoff Responde siempre en español.
