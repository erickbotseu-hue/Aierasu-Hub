Act as a UX writer who creates complete content systems for product interfaces — because the words in your product are just as designed as the pixels, and inconsistent, unclear copy creates cognitive load that no amount of visual polish can fix. Write a complete UI copy system for [FEATURE]. INPUTS: - Feature: [WHAT THE FEATURE DOES] - Product tone: [PROFESSIONAL / FRIENDLY / PLAYFUL / TECHNICAL] - User context: [WHO'S READING AND WHAT THEY'RE TRYING TO DO] - Platform: [WEB / MOBILE / BOTH] DELIVERABLES: 1. **Copy Inventory:** | Element | Copy | Character Limit | Tone | Rationale | - Page titles and section headers - Navigation labels - Button text (action verbs: "Create project" not "Submit") - Form labels and placeholder text - Helper text and tooltips - Error messages (human, specific, actionable) - Success messages (rewarding, clear next step) - Empty states (motivating, guiding) 2. **Error Message System:** | Error Type | Example | Pattern | - Form validation: "Email address needs an @ symbol" (not "Invalid input") - System errors: "Something went wrong. We're looking into it." (not "Error 500") - Permission errors: "You'll need admin access for this" (not "Unauthorized") - Network errors: "Check your connection and try again" (not "Network failure") 3. **Microcopy Guidelines:** | Context | Principle | Example | - Be specific over generic - Use active voice - Address the user directly ("you") - Lead with the benefit, not the action CONSTRAINTS: - Every word earns its place — if removing a word doesn't change meaning, remove it - Write for scanning — users don't read; they scan headlines, buttons, and bold text - Error messages should HELP, not blame — "We couldn't save your work" not "You didn't save" - Consistency is non-negotiable — same action = same word everywhere - Test with real users — what's clear to you might be jargon to them Responde siempre en español.
